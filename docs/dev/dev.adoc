== Overview

_fs_ is a lightweight, multi-version JDK support, high-performance, zero-dependency tool lib for java.

=== Project Architecture

_fs_ follows a modular architecture with clear separation of concerns:

* <<chapter-dependencies>>: Dependency management platform (BOM)
* <<chapter-jsr305>>: JSR-305 annotations implementation
* <<chapter-annotation>>: Custom annotations for code analysis and validation
* <<chapter-asm>>: Embedded ASM framework for bytecode manipulation
* <<chapter-core>>: Main library with comprehensive utilities and interfaces
* <<chapter-all>>: Aggregated jar containing all modules (zero-dependency)
* <<chapter-internal>>: Internal utilities for testing
* <<chapter-tests>>: Test suite and benchmarks

=== Design Principles

_fs_ is built on the following design principles:

* **Zero Dependency**: Minimal external dependencies, self-contained implementations to ensure portability and reduce dependency conflicts
* **Multi-Version Support**: Automatic JDK version detection and optimal implementation selection for backward compatibility and forward optimization
* **High Performance**: Optimized implementations with benchmark validation to ensure efficient execution
* **Modularity**: Clear module boundaries and responsibilities for maintainability and scalability
* **Extensibility**: Well-defined interfaces for custom implementations to support extensibility
* **Consistency**: Unified API design and naming conventions across modules
* **Reliability**: Comprehensive test coverage and error handling to ensure robust functionality
* **Simplicity**: Intuitive APIs and clear documentation to reduce learning curve

=== Development Guidelines

When contributing to _fs_, follow these guidelines:

* **Package Structure**: Use the existing package structure and naming conventions to maintain consistency
* **Multi-Version Support**: Follow the multi-version JDK support pattern using `FsLoader` for optimal implementation selection
* **Testing**: Write comprehensive tests for new functionality and maintain 100% test coverage
* **Null Safety**: Use the built-in annotations (`@Nonnull`, `@Nullable`) for null safety and code analysis
* **Code Style**: Follow the existing code style and naming conventions
* **Documentation**: Provide clear and comprehensive documentation for all public APIs
* **Performance**: Optimize critical paths and validate performance with benchmarks
* **Error Handling**: Use the built-in exception hierarchy and handle errors appropriately
* **Dependency Management**: Avoid adding external dependencies unless absolutely necessary
* **Backward Compatibility**: Maintain backward compatibility when introducing changes

=== Coding Conventions

* **Naming Conventions**:
  * Class names: PascalCase
  * Method names: camelCase
  * Variable names: camelCase
  * Constant names: UPPER_SNAKE_CASE
  * Package names: lowercase with dots as separators

* **Code Formatting**:
  * Use 4 spaces for indentation
  * Keep lines under 120 characters
  * Use blank lines to separate logical blocks
  * Use curly braces even for single-line blocks

* **Documentation**:
  * Use Javadoc for all public classes and methods
  * Provide clear and concise descriptions
  * Include examples where appropriate
  * Document parameters, return values, and exceptions

* **Error Handling**:
  * Use specific exception types when possible
  * Provide meaningful error messages
  * Catch exceptions at the appropriate level
  * Use `Fs.uncheck` for checked exception handling when appropriate

[NOTE]
====
* _fs-dependencies_ should be published as a `bom` (`<packaging>pom</packaging>`);
* _fs-jsr305_, _fs-annotation_, _fs-asm_, _fs-core_ and _fs-all_ should be published as jars;
* _fs-internal_ and _fs-tests_ should not be published;
====

_fs_ uses `gradle` to build, and it needs multi JDK versions to build: from `1.8` to `17`.

[NOTE]
====
The root `gradle` properties file link:../gradle.properties[gradle.properties] can configure the multi JDK versions like:

[source,properties]
----
org.gradle.java.installations.paths=\
  ~/.jdks/azul-1.8.0_462,\
  ~/.jdks/azul-17.0.17,\
  ~/.jdks/openjdk-25,
----

Add/Change this property to your local JDK paths, or let `gradle` automatically download required JDK versions without manual configuration.
====

== _fs-annotation_

src: link:../fs-annotation/[fs-annotation]

_fs-annotation_ is the annotation module of _fs_, should be published as a jar.
It depends on _fs-jsr305_.

=== Purpose

Provides custom annotations for code analysis, validation, and static analysis support.

=== Key Annotations

* **Null Safety**: `@Nonnull`, `@Nullable` for null pointer prevention
* **Immutability**: `@Immutable` for immutable types
* **Parameter Handling**: `@RetainedParam` for parameter retention
* **Visibility**: `@NonExported` for non-exported packages

=== Usage Guidelines

* Use `@Nonnull` and `@Nullable` consistently for method parameters and return types
* Apply `@Immutable` to value objects and data classes
* Use package-level annotations for default nullability settings

=== Integration

These annotations are used throughout _fs-core_ and can be used in external projects for consistent code analysis.

[[chapter-asm]]
== _fs-asm_

src: link:../fs-asm/[fs-asm]

_fs-asm_ is the internal asm (7.1) for _fs_, should be published as a jar.

=== Purpose

Embedded ASM framework for bytecode manipulation, providing dynamic programming capabilities without external dependencies.

=== Key Features

* **Zero Dependency**: Complete ASM implementation with no external dependencies
* **Package Isolation**: All classes in `space.sunqian.fs.asm` package
* **Version Compatibility**: ASM 7.1 feature set

=== Core Components

* **Class Manipulation**: `ClassReader`, `ClassWriter`
* **Method Handling**: `MethodVisitor`, `FieldVisitor`
* **Bytecode Analysis**: `AnnotationVisitor`, `Type`
* **Framework Support**: Complete ASM opcodes and utilities

=== Usage in _fs_

* Used by _fs-core_ for dynamic proxy and AOP functionality
* Enables bytecode generation for performance optimization
* Provides reflection alternatives for high-performance scenarios

=== Development Notes

* This is a direct copy of ASM framework with package relocation
* No modifications to original ASM functionality
* Used internally by _fs-core_ dynamic programming features

[[chapter-core]]
== _fs-core_

src: link:../fs-core/[fs-core]

_fs-core_ is main part of _fs_, should be published as a jar.
It depends on _fs-annotation_.

=== Architecture Overview

_fs-core_ is organized into functional packages with clear responsibilities:

* **Root Package** (`space.sunqian.fs`): Core utilities and global access point
  * **Fs Class**: The core utility class providing comprehensive functionality

* **Base Utilities** (`space.sunqian.fs.base`): Fundamental utilities for common operations
  * **Common Base**: `Checker`, `FsLoader` for core functionality
  * **Language Extensions**: `lang` package for language enhancements
  * **Functional Extensions**: `function`, `value`, `option` packages
  * **Base Utility Modules**: `bytes`, `chars`, `date`, `exception`, `logging`, `math`, `number`, `process`, `random`, `string`, `system`, `thread`

* **Cache** (`space.sunqian.fs.cache`): Lightweight caching implementation
  * **SimpleCache**: Soft/weak reference-based caching
  * **CacheFunction**: Map-based cache functionality
  * **Usage Example**: `SimpleCache.ofSoft()`, `SimpleCache.ofWeak()`

* **Collections & Data** (`space.sunqian.fs.collect`, `space.sunqian.fs.data`): Enhanced collection and data processing
  * **Collection Utilities** (`collect`): Array, List, Set, Map operations
    * **Kits**: CollectKit, ListKit, SetKit, MapKit, ArrayKit, StreamKit
    * **Interfaces**: ArrayOperator
  * **Data Processing** (`data`): JSON, Properties handling
    * **Core Interfaces**: ByteData, CharData, ByteDataParser, CharDataParser, ByteDataFormatter, CharDataFormatter
    * **Utilities**: DataKit
    * **Subpackages**: json, properties

* **Concurrency** (`space.sunqian.fs.concurrent`): Concurrent programming utilities
  * **FutureKit**: Future-related utilities

* **I/O Operations** (`space.sunqian.fs.io`): Byte/char processing and file operations
  * **Core Utilities**: IOKit, BufferKit
  * **Data Reading**: ByteReader, CharReader, ByteSegment, CharSegment
  * **Data Transformation**: ByteProcessor, ByteTransformer, CharProcessor, CharTransformer
  * **Stream Implementation**: DoReadStream, DoReadReader, DoWriteStream, DoWriteWriter
  * **Subpackages**: file (FileKit, FileRef), communicate (Channel interfaces)

* **Networking** (`space.sunqian.fs.net`): HTTP, TCP, UDP with adaptive implementations
  * **Core Utilities**: NetKit
  * **Core Interfaces**: NetClient, NetServer
  * **Subpackages**:
    * **http**: HTTP client with JDK version-adaptive implementations
    * **tcp**: TCP client/server with builder patterns and handlers
    * **udp**: UDP sender/server with datagram communication

* **Object Manipulation** (`space.sunqian.fs.object`): Schema, building, conversion, and pooling
  * **Core Utilities**: ObjectKit
  * **Core Exception**: ObjectException
  * **Subpackages**:
    * **schema**: Object schema definition and parsing
    * **builder**: Object building with builder patterns
    * **convert**: Object conversion and property copying
    * **pool**: Object pooling interfaces
    * **annotation**: Annotations for object-related operations

* **Dynamic Programming** (`space.sunqian.fs.dynamic`, `space.sunqian.fs.invoke`): Proxy, AOP, and reflection
  * **Core Utilities** (`dynamic`): DynamicKit, DynamicClassLoader
  * **Subpackages** (`dynamic`):
    * **proxy**: JDK and ASM-based dynamic proxy
    * **aspect**: Aspect-oriented programming (AOP) support
  * **Invocation** (`invoke`): Reflection and method-handle based invocation

* **Dependency Injection** (`space.sunqian.fs.di`): Lightweight DI container
  * **Core Interfaces**: DIContainer, DIComponent, DIAspectHandler
  * **Utilities**: DIKit
  * **Key Features**:
    * Component lifecycle management
    * Dependency resolution and injection
    * AOP integration
    * Parent-child container hierarchy
  * **Usage Example**: Builder pattern for container creation with component registration

* **Reflection** (`space.sunqian.fs.reflect`): Reflection utilities and type handling
  * **Utilities**: TypeKit, ClassKit
  * **Classes**: TypeRef
  * **Key Features**:
    * Type compatibility checking
    * Class manipulation utilities
    * Type reference handling

* **Third-party Integration** (`space.sunqian.fs.third`): Integration with third-party libraries
  * **Utilities**: ThirdKit
  * **Subpackages**:
    * **asm**: Embedded ASM framework for bytecode manipulation
    * **protobuf**: Protocol Buffers support for object conversion

* **Utilities** (`space.sunqian.fs.utils`): Codec, event bus, JDBC, versioning
  * **Subpackages**:
    * **codec**: Base64, Hex, Crypto, Digest utilities
    * **eventbus**: Publish-subscribe communication
    * **jdbc**: Database operations and SQL handling
    * **version**: Semantic versioning support

=== Multi-Version JDK Support

The codes from `JDK8` to `JDK17` are written here, and the build script automatically compiles into the corresponding version of the class file. `FsLoader` is used to distinguish between different implementation versions:

[source,java]
----
interface HttpCallerService {

    @Nonnull
    HttpCallerService INST = FsLoader.loadImplByJvm(HttpCallerService.class, 11);

    @Nonnull
    HttpCaller newCaller(
        int bufSize,
        @Nonnull Proxy proxy
    ) throws HttpNetException;
}
----

=== Key Design Patterns

* **Factory Pattern**: Used extensively for creating instances with version-specific implementations
* **Builder Pattern**: Network clients and object builders use fluent builder interfaces
* **Strategy Pattern**: Different implementations selected based on runtime environment
* **Adapter Pattern**: JDK version compatibility through adapter implementations

=== Development Guidelines

* Place kits, interfaces, and implementations in appropriate packages
* Use `FsLoader` for multi-version JDK support
* Follow the established naming conventions and package structure
* Write comprehensive tests for all new functionality
* Maintain interface-back-implementation pattern for extensibility

=== Performance Considerations

* Optimize hot paths with benchmark validation
* Use appropriate data structures for performance-critical operations
* Leverage ASM-based implementations for dynamic programming features
* Implement efficient caching strategies where appropriate

== Build System

=== Gradle Configuration

_fs_ uses a custom Gradle build system with the following features:

* **Multi-Module Support**: Clear separation of module responsibilities
* **Multi-JDK Build**: Support for JDK 8 through 17
* **Custom Plugins**: `fs` and `fs-publish` plugins for build automation
* **Dependency Management**: Centralized dependency versions

=== Build Tasks

Key build tasks available:

* `gradle build` - Build all modules
* `gradle test` - Run all tests
* `gradle jmh` - Run JMH benchmarks
* `gradle javadoc` - Generate aggregated Javadoc
* `gradle jacocoTestReport` - Generate coverage reports

=== Publishing

Publishing configuration includes:

* **Maven Central**: Support for publishing to Maven repositories
